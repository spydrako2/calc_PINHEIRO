# Story 1.4: Parser SPPREV Aposentados

**Status:** InReview
**Story ID:** 1.4
**Epic:** 1 - Core Engine
**Priority:** High
**Complexity:** High

## User Story

Como usuario,
eu quero que o sistema extraia corretamente todos os dados de holerites SPPREV de aposentados,
para que eu possa processar holerites de servidores aposentados.

## Acceptance Criteria

1. Parser SPPREV Aposentados implementado seguindo a interface base_parser
2. Extrai cabecalho especifico: nome, entidade, cargo, competencia, regime retribuicao, beneficio, percentual aposentadoria, CPF, numero beneficio, tipo folha, banco/agencia/conta, nivel
3. Extrai itens de linha com: codigo (formato XXXXXX sem ponto), denominacao, natureza, quantidade, unidade, periodo, e valores separados em colunas Vencimento e Descontos
4. Extrai totais do rodape: base IR, base redutor, base contrib prev, total vencimentos, total descontos, total liquido
5. Template detector identifica pela presenca de "DIRETORIA DE BENEFICIOS SERVIDORES" no cabecalho
6. Testes com holerites SPPREV aposentados reais

## Dev Notes

- Reutilizar `BaseParser` da Story 1.3 - não reimplementar
- SPPREV Aposentados: padrões similares a DDPE mas com labels e estrutura diferentes (cabecalho com mais campos específicos)
- Usar `pdf_reader.PaginaExtraida` como input, retornar `data_model.Holerite`
- Regex patterns devem ser case-insensitive e tolerantes a espaçamento
- Valores monetários em formato brasileiro: 1.234,56 (ponto = milhares, vírgula = centavos)
- Códigos de verba: formato XXXXXX sem ponto (diferente de DDPE que usa XX.XXX)
- Usar `normalizer.CodigoVerbaNormalizer` para padronização

## Testing

- Usar PDFs reais de `docs/referencias/` (SPPREV aposentados)
- Validar que todos os campos obrigatórios são extraídos
- Testar com holerites de múltiplas páginas
- Testar edge cases: proventos com valores zero, descontos faltantes, etc
- Comparar padrões de extração com DDPE para consistência
